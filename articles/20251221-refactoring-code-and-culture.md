## はじめに

この記事は [LITALICO Advent Calendar 2025](https://qiita.com/advent-calendar/2025/litalico) シリーズ1の21日目の記事です。私が LITALICO に入社して約1年が経ちました。次の年が始まる前に、この1年間どんなことに取り組み、何を考えていたのかをまとめてみます。

## 開発者体験(DX)の向上に取り組む

LITALICO では新旧様々なサービスを提供しているのですが、私が配属されたチームのプロダクトは10年以上の歴史があり、ベテランメンバーも多い環境でした。しかし、長い運用の中でツール類が老朽化しており、メンテナンスの手が回っていない状態でした。そこで、個人のタスク消化だけでなく、チーム全体の開発効率を上げるための手入れになるべく時間を割くようにしました。

### Vagrant から Docker Compose へ移行した

開発環境は Windows 前提の Vagrant/VirtualBox を使った構成しかなく、Mac ユーザーは各々が自力で環境構築をしている状態でした。データベースはローカル環境を作らずに AWS に専用サーバーを立てているという話も聞きました。これでは新しく入る人が困り続けるだろうと感じたため、開発環境の再構成に着手しました。

Oracle DB のコンテナ化や、古い Ruby バージョンに起因するトラブルなどがありました。起動したこともないアプリの環境構築は骨の折れる作業でした。ここでつまずくと、何もできず退職なんてこともあるんじゃないかと怯えながら、たくさん Slack で質問していたのを覚えています。それでも、なんとか完了して、手元で動かせるようになった時はホッとしました。私の作った環境構築手順が、後から参加してきた人たちにも確かに役立っているのが目に見えて嬉しかったです。

### CI を作ってメンテナンスした

CI がなかったため、GitHub Actions を導入し、プルリクエスト作成時に単体テストを全件実行するようにしました。当初はローカルで失敗するテストの修正に追われました。中でも、テストコードが間違っているケースへの対応が大変でした。プロダクトの理解度が低い状態では、どういう振る舞いが正しいのかわかりません。無効なテストとしてフラグを立てたり、仕様について質問したり、一つ一つ対処していくのは長い道のりでした。

もたついているうちに、CI で動かないテストコードがマージされたり、不安定なテスト (Flaky Test) が発見されたりとトラブルは尽きませんでした。特に CI 環境下での Oracle コンテナの制御に苦戦しました。調べても情報が少なく、ライセンスの問題、イメージサイズの大きさ、CPUアーキテクチャの問題など、非常に歩みが遅かったです。MySQL や PostgreSQL のようなエコシステムの充実した技術がいかに便利かを痛感しました。試行錯誤の末 CI 導入までこぎつけ、実際に「これのおかげで不具合に気づけた」という声も二、三回耳にしました。無事に結果が出せてよかったです。

ただ動かすだけではなく、チューニングや継続的なメンテナンスもしました。テスト実行は直列実行では60分ほどかかってしまうため、並列化やキャッシュを導入して、遅くても15分程度で完了できるようにしました。その後も、ランナーのアーキテクチャ変更、スタイルチェックの追加、カバレッジレポートの追加など、少しずつ改善を続けています。

### 連携 API 開発

他のプロダクトと連携するための REST API を作ることになりました。技術的には OpenAPI を使って Redoc でドキュメントを書き出すようにしました。OpenAPI ならコード自動生成もできるし、クライアントにとっては利用しやすいのではないかと思います。

この頃、ちょうど AI が全社的に提供され始めたので使ってみました。既存画面のフォームの HTML を取り出して AI に渡して「同じインターフェースを持つ OpenAPI の仕様書を作成して欲しい」と指示したところ、それなりの良い出力が得られました。こうしたやり方は他のプロジェクトでも役に立つかもしれません。

## 開発プロセスの改善

リモートワーク中心の環境は働きやすい反面、「困った時の助け合い」や「知識の共有」といった、個人のタスクを超えた動きが少ないように感じていました。個々に割り振られた仕事はしっかりとこなしている一方で、その線を超えて能動的に動く人は限られていると感じました。

### 振り返りの企画

チーム全体に目を向けるきっかけを作るため、振り返りイベントを主催しました。当初は「チームの課題解決」を目標に掲げました。課題を見つけ、それにどのように取り組むかという枠組み自体は隙なく計画していたつもりでした。しかし、4ヶ月続けてみても、大きな手応えは得られませんでした。アイデアが出にくかったり、出てきたアイデアも実践されずに停滞したり温度差を感じました。このまま続けてもうまくいきそうな気配がないと考え、路線変更をすることにしました。

課題を見つけて解決することよりも、一歩手前に必要なことがあるのではないか。そう考え、新しい目標を「チームメンバーの信頼関係を作る」に切り替えました。長く働いているベテランが多いとはいえ、リモートワークでは会話の機会は限られています。とりあえず4人グループに分かれて、話す機会を作るだけでも効果があるのではないか、と考えました。

3ヶ月続けてみて、この方針の方が今のチームに合っていると感じました。リモートワークで失われた雑談の機会にぴったりはまっているように思います。ささやかな日常や、ちょっとした困りごとを、リラックスして話す時間になりました。

後半には、困りごとを膨らませて、チームとして向き合ってみる時間もとっています。ここでは以前の失敗を踏まえて、「課題の解決」ではなく「課題への共感」をゴールにしました。具体的な解決策や責任については考えません。共感を入り口にして「改善したい」というモチベーションが高まったらアクションすれば良い、という考えです。本当に重要なことが見つかったら、リーダーやマネージャーが後押しする予定です。

### スクラムマスター

上記の振り返りを企画している中で、どうせなら PDCA サイクルを作って、開発期間をスプリントに区切ってみたいという話がマネージャから出てきました。隣のチームでよくなった実績があるので、私たちのチームにも導入してみたいとのことでした。私以外のスクラム経験者が少なく、指揮を取れる人がいなかったため、その場の流れで私がスクラムマスターをやることになりました。実のところ、そこまで気乗りしていたわけではないのですが、自由な裁量のもとで進めさせてもらったので、やっているうちに面白くなってきました。

スクラムの導入に関しては、大きく開発プロセスを変えるのではなく、少しずつ変える方針にしました。すでにベテランが長年続けてきた開発スタイルがあるので、それを邪魔したくないと思ったからです。スクラムイベントも重要なところだけに絞り、できるだけメンバーの負担が増えにくいようにしました。

結果としては、大きな混乱もなくチームに導入することができました。劇的に何かが変わったというわけではないのですが、月々の成果物がどれくらいで、見通しがどれくらいなのかは見えやすくなりました。コミュニケーションの機会が増え、リーダーが独断で決めるのではなく、チームで判断できる場面も増えました。こうした変化が、リリースプロセスの安定化につながっていると感じています。まだ手探りの部分はありますが、良い傾向ではないかと思います。

## 次にやりたいこと

カバレッジがまだまだ低いので、AI を使って大量にテストコードを作るのは意外と良い戦略なのではないかと思います。カバレッジが高くなれば Ruby のバージョンアップやリファクタリングのリスクが下がって、今後の仕事も進めやすくなるはずです。

別の視点では、繁忙期にお問い合わせの電話が繋がりにくくなってしまうという点を改善できないかと思っています。もちろんコールセンターの人員拡大といった正攻法での改善も考えられますが、開発部目線での改善点もあるはずです。例えば、問い合わせが多い機能を分析して、迷わないUX/UIを作ること。それから、お問い合わせの解決を早めるために、コールセンターと開発部のコミュニケーションをスムーズにするためのツールを導入すること。そう簡単な話ではないと思いますが、何か見落としていることがあるかもしれません。一度振り返ってみると新しい発見があるのではないかと思いました。

こうした意欲がある一方で、私生活の方もバランスをとっていかないといけません。子供が成長するにつれて、重い障害を持っていることがわかりました。言葉では尽くせないけれど、喜びと同じかそれ以上のつらさを毎週のように感じています。しかし、この痛みを感じるほど、同じように支えの足りない人たちに優しくしたいという気持ちになります。そんな気持ちで仕事にも向き合っていけたら良いなと思います。

## おわりに

入社して働く場所や環境が変わると、どうしても「前の会社ではこうだった」「あのやり方ならもっとスムーズなのに」という違いが目についてしまうものです。正直なところ、当初は「非効率な部分を一掃するために、大胆な改革や破壊的な変更が必要なんじゃないか」と短絡的に考えてしまう場面もありました。

しかし、プロセスやツールを変えることは、そこで日々働いているチームメンバーにとっては大きな負担でもあります。新しいやり方を覚える労力はもちろん、「これまでの積み重ねが否定される」と感じさせてしまうかもしれません。何より、まだ十分な信頼関係も築けていない状態で私が権限を振るい、強引に変化を進めたとしたら、今のチームが持っている良さや安心感まで壊してしまうと思います。だからこそ、チームとの間に温度差を作らないようなゆっくりとした変化を心がけました。

もちろん、激しい変化が求められる場面もあると思うんですが、どうも私はゆっくりしたやり方が好みなようです。このまま少しずつ変化を積み重ねて「気がついたら、いつの間にか良いチームになっていたね」と言えるような、なめらかな進化を目指してやっていきたいと思います。
